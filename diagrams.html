Lesson Plan

A Python MCP server providing read-only access to brokerage data via the SnapTrade API. ~500 lines total — 7 lessons to full understanding.

Lesson 1: Project Structure & Setup

Understand how the project is organized and how to install it.

Files to read

  • pyproject.toml — dependencies, entry points, build config
  • snaptrade_mcp/__init__.py — version metadata
  • snaptrade_mcp/__main__.py — module entry point

Key concepts

  • Python packaging with hatchling
  • Two dependencies: mcp[cli] (MCP framework) and snaptrade-python-sdk (API client)
  • Two entry points: snaptrade-mcp CLI command and python -m snaptrade_mcp
  • Requires Python 3.10+
Exercise: Install the project with pip install -e . (editable mode)
Diagrams: 1. Architecture

Lesson 2: MCP Server Basics

Understand how FastMCP works and how the server is initialized.

File

  • snaptrade_mcp/server.py (lines 1–30)

Key concepts

  • FastMCP — simplified MCP server class
  • STDIO transport — server communicates via stdin/stdout (not HTTP)
  • @mcp.tool() — decorator to register tools that AI agents can call
  • @mcp.resource() — decorator to expose static data endpoints
  • @mcp.prompt() — decorator for prompt templates
  • main() calls mcp.run() to start the server
Diagrams: 1. Architecture 2. Components 3. Class/Module

Lesson 3: Authentication & Configuration

Understand the two-layer auth model.

File

  • snaptrade_mcp/server.py — _get_client() and _get_user() functions

Key concepts

  • Layer 1 — Developer credentials (env vars): SNAPTRADE_CLIENT_ID + SNAPTRADE_CONSUMER_KEY
  • Layer 2 — User credentials (local file): ~/.snaptrade/config.json with user_id + user_secret
  • _get_client() reads env vars and creates a SnapTrade SDK instance
  • _get_user() reads the local config file
  • snaptrade_setup() tool creates user credentials on first run and generates a brokerage auth URL
  • Security: config file created with chmod 600, credentials never exposed in tool responses
Diagrams: 6. Setup Sequence 8. Auth Layers

Lesson 4: Tools — The Core Functionality

Understand how MCP tools work and what this server exposes.

File

  • snaptrade_mcp/server.py — all @mcp.tool() functions

10 tools (recommended study order)

  • snaptrade_check_status() — health check, good starting point
  • snaptrade_setup() — creates user config, generates auth URL
  • snaptrade_list_accounts() — lists connected brokerage accounts
  • snaptrade_get_balance(account_id) — cash balances per account
  • snaptrade_get_positions(account_id) — stock/ETF holdings
  • snaptrade_get_orders(account_id, status) — order history with filtering
  • snaptrade_get_activities(account_id) — transaction log
  • snaptrade_portfolio_summary() — aggregates all accounts (most complex)
  • snaptrade_search_symbols(query) — search stocks/ETFs
  • snaptrade_list_brokerages() — supported brokerages

Pattern

  • Every tool: get client → get user → call SDK → serialize response → return JSON string
Diagrams: 3. Class/Module 4. Data Flow 5. Tool Call Sequence 7. Portfolio Summary 11. Tool Dependency Map

Lesson 5: Data Serialization & Error Handling

Understand how SDK responses are converted to MCP-friendly output.

File

  • snaptrade_mcp/server.py — _serialize(), _format_response(), _clean_error()

Key concepts

  • SDK returns complex objects — _serialize() recursively converts them to dicts
  • Handles objects with .body or .to_dict() methods
  • _format_response() wraps serialization + JSON formatting
  • _clean_error() extracts concise messages from verbose SDK exceptions
  • portfolio_summary gracefully continues if individual account calls fail
Diagrams: 4. Data Flow 9. Serialization Pipeline 10. Error Handling

Lesson 6: Resources & Prompts

Understand the non-tool MCP features.

File

  • snaptrade_mcp/server.py — @mcp.resource() and @mcp.prompt() sections

Resources (2)

  • snaptrade://status — exposes health check as a resource
  • snaptrade://brokerages — exposes brokerage list as a resource

Prompts (2)

  • analyze_portfolio() — instructs AI to do portfolio analysis (allocation, concentration risk, diversification)
  • account_summary() — instructs AI to create a concise account table
Diagrams: 2. Components

Lesson 7: Running & Testing

Run the server and test it end-to-end.

Steps

  • Get SnapTrade API credentials from https://snaptrade.com
  • Set env vars: export SNAPTRADE_CLIENT_ID=... SNAPTRADE_CONSUMER_KEY=...
  • Run: python -m snaptrade_mcp (starts STDIO server)
  • Or add to Claude Code: claude mcp add snaptrade -s user -e SNAPTRADE_CLIENT_ID=... -e SNAPTRADE_CONSUMER_KEY=... -- snaptrade-mcp
  • Test by asking Claude to call snaptrade_check_status

    participant SDK as SnapTrade SDK
    participant API as SnapTrade API
    participant FS as File System

    end

    MCP->>SDK: login_snap_trade_user(user_id, user_secret)
    SDK->>API: POST /snapTrade/login
    API-->>SDK: {redirectURI}
    SDK-->>MCP: login response
    MCP->>BR: webbrowser.open(redirectURI)
    MCP-->>AI: {"status": "opened", "message": "..."}

7. Sequence Diagram — Portfolio Summary

Lesson 4

The most complex tool — aggregates data across all accounts with graceful error handling.

sequenceDiagram
    participant AI as AI Client
    participant MCP as MCP Server
    participant SDK as SnapTrade SDK

    AI->>MCP: tool_call("snaptrade_portfolio_summary")
    MCP->>SDK: list_user_accounts()
    SDK-->>MCP: [Account A, Account B]

    loop For each account
        MCP->>SDK: get_user_account_balance(account_id)
        alt Success
            SDK-->>MCP: balance data
        else Error
            SDK-->>MCP: exception
            MCP->>MCP: _clean_error() → store error msg
        end

        MCP->>SDK: get_user_account_positions(account_id)
        alt Success
            SDK-->>MCP: positions data
        else Error
            SDK-->>MCP: exception
            MCP->>MCP: _clean_error() → store error msg
        end
    end

    MCP->>MCP: Assemble portfolio array
    MCP->>MCP: _format_response()
    MCP-->>AI: {"portfolio": [...], "account_count": 2}

9. Serialization Pipeline

Lesson 5

How _serialize() recursively converts SDK objects to JSON-friendly dicts.

flowchart TD
    IN["Input: SDK Response Object"] --> A{"Has .body?"}
    A -- Yes --> B["Unwrap: obj = obj.body"]
    B --> C
    A -- No --> C{"Is list?"}
    C -- Yes --> D["Recurse: [_serialize(item) for item in obj]"]
    C -- No --> E{"Has .to_dict()?"}
    E -- Yes --> F["Call obj.to_dict()"]
    E -- No --> G{"Has __dict__?"}
    G -- Yes --> H["Extract: {k:v for k,v
in __dict__
if not k.startswith('_')}"]
    G -- No --> I["Return obj as-is
(primitive type)"]

    F --> J["_format_response()"]
    H --> J
    I --> J
    D --> J
    J --> K["json.dumps(data,
indent=2,
default=str)"]
    K --> OUT["JSON String
→ AI Client"]

    style IN fill:#ffedd5,color:#7c2d12,stroke:#ea580c,stroke-width:2px

12. Server State Diagram

Lesson 7

The lifecycle states of the MCP server and user configuration.

stateDiagram-v2
    [*] --> Unconfigured: Server starts

    Unconfigured --> NoEnvVars: ENV vars missing
    Unconfigured --> NoUserConfig: ENV vars set,
no config.json

    NoEnvVars --> Unconfigured: User sets ENV vars

    NoUserConfig --> Registering: snaptrade_setup called
    Registering --> UserRegistered: API returns user_secret
    UserRegistered --> ConfigSaved: Write config.json
(chmod 600)
    ConfigSaved --> AuthorizingBrokerage: Open browser
redirect URL

    AuthorizingBrokerage --> Ready: User authorizes
brokerage
    AuthorizingBrokerage --> ConfigSaved: User cancels

    Ready --> ServingTools: Any tool called
    ServingTools --> Ready: Response returned
    ServingTools --> ErrorState: API error
    ErrorState --> Ready: Next call succeeds

    Ready --> [*]: Server process exits
