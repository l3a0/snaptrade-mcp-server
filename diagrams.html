Lesson Plan

A Python MCP server providing read-only access to brokerage data via the SnapTrade API. ~900 lines total — 7 lessons to full understanding.

Lesson 1: Project Structure & Setup

Understand how the project is organized and how to install it.

Files to read

  • pyproject.toml — dependencies, entry points, build config
  • snaptrade_mcp/__init__.py — version metadata
  • snaptrade_mcp/__main__.py — module entry point
  • snaptrade_mcp/oauth_provider.py — in-memory OAuth 2.0 provider for HTTP transport
  • snaptrade_mcp/snaptrade_client.py — SnapTrade SDK import wrapper

Key concepts

  • Python packaging with hatchling
  • Two dependencies: mcp[cli] (MCP framework) and snaptrade-python-sdk (API client)
  • Two entry points: snaptrade-mcp CLI command and python -m snaptrade_mcp
  • Requires Python 3.10+
Exercise: Install the project with pip install -e . (editable mode)
Diagrams: 1. Architecture

Lesson 2: MCP Server Basics

Understand how FastMCP works and how the server is initialized.

File

  • snaptrade_mcp/server.py (lines 1–120: imports, OAuth init, FastMCP setup)

Key concepts

  • FastMCP — simplified MCP server class
  • STDIO transport (default) — server communicates via stdin/stdout; HTTP transport (streamable-http) — for remote clients like ChatGPT via OAuth 2.0
  • @mcp.tool() — decorator to register tools that AI agents can call
  • @mcp.resource() — decorator to expose static data endpoints
  • @mcp.prompt() — decorator for prompt templates
  • main() parses --transport, --host, --port args and calls mcp.run(transport=...)
Diagrams: 1. Architecture 2. Components 3. Class/Module

Lesson 3: Authentication & Configuration

Understand the three-layer auth model.

Files

  • snaptrade_mcp/server.py — _get_client(), _get_user(), OAuth init, AuthSettings, TransportSecuritySettings
  • snaptrade_mcp/oauth_provider.py — SimpleOAuthProvider class (in-memory OAuth 2.0 server)

Key concepts

  • Layer 1 — Developer credentials (env vars): SNAPTRADE_CLIENT_ID + SNAPTRADE_CONSUMER_KEY
  • Layer 2 — User credentials (local file): ~/.snaptrade/config.json with user_id + user_secret
  • Layer 3 — OAuth credentials (env vars, HTTP transport only): SNAPTRADE_OAUTH_CLIENT_ID, SNAPTRADE_OAUTH_CLIENT_SECRET, SNAPTRADE_OAUTH_REDIRECT_URI, SNAPTRADE_PUBLIC_URL
  • _get_client() reads env vars and creates a SnapTrade SDK instance
  • _get_user() reads the local config file
  • SimpleOAuthProvider handles OAuth 2.0 auth code flow with PKCE, auto-approve, token rotation
  • snaptrade_setup() tool creates user credentials on first run and generates a brokerage auth URL
  • Security: config file is chmod 600, PKCE required, DNS rebinding protection via TransportSecuritySettings
Diagrams: 6. Setup Sequence 8. Auth Layers 13. OAuth Flow 14. Transport Selection

Lesson 4: Tools — The Core Functionality

Understand how MCP tools work and what this server exposes.

File

  • snaptrade_mcp/server.py — all @mcp.tool() functions

10 tools (recommended study order)

  • snaptrade_check_status() — health check, good starting point
  • snaptrade_setup() — creates user config, generates auth URL
  • snaptrade_list_accounts() — lists connected brokerage accounts
  • snaptrade_get_balance(account_id) — cash balances per account
  • snaptrade_get_positions(account_id) — stock/ETF holdings
  • snaptrade_get_orders(account_id, status) — order history with filtering
  • snaptrade_get_activities(account_id) — transaction log
  • snaptrade_portfolio_summary() — aggregates all accounts (most complex)
  • snaptrade_search_symbols(query) — search stocks/ETFs
  • snaptrade_list_brokerages() — supported brokerages

Pattern

  • Every tool: get client → get user → call SDK → serialize response → return JSON string

Lesson 5: Data Serialization & Error Handling

Understand how SDK responses are converted to MCP-friendly output.

File

  • snaptrade_mcp/server.py — _serialize(), _format_response(), _clean_error()

Key concepts

  • SDK returns complex objects — _serialize() recursively converts them to dicts
  • Handles objects with .body or .to_dict() methods
  • _format_response() wraps serialization + JSON formatting
  • _clean_error() extracts concise messages from verbose SDK exceptions
  • portfolio_summary gracefully continues if individual account calls fail

Resources (2)

  • snaptrade://status — exposes health check as a resource

Prompts (2)

Lesson 7: Running & Testing

Run the server and test it end-to-end.

Steps

  • Get SnapTrade API credentials from https://snaptrade.com
  • Set env vars: export SNAPTRADE_CLIENT_ID=... SNAPTRADE_CONSUMER_KEY=...
  • STDIO mode: python -m snaptrade_mcp (default, for Claude/Cursor)
  • HTTP mode: snaptrade-mcp --transport streamable-http (requires OAuth env vars + SNAPTRADE_PUBLIC_URL)
  • Or add to Claude Code: claude mcp add snaptrade -s user -e SNAPTRADE_CLIENT_ID=... -e SNAPTRADE_CONSUMER_KEY=... -- snaptrade-mcp
  • Test by asking Claude to call snaptrade_check_status

How _serialize() recursively converts SDK objects to JSON-friendly dicts.

flowchart TD

    A -- Yes --> B["Unwrap: obj = obj.body"]
    B --> C
    A -- No --> C{"Is list?"}

    E -- No --> G{"Has __dict__?"}
    G -- Yes --> H["Extract: {k:v for k,v

    G -- No --> I["Return obj as-is
(primitive type)"]

    F --> J["_format_response()"]
    H --> J
    I --> J
    D --> J
    J --> K["json.dumps(data,
indent=2,
default=str)"]
    K --> OUT["JSON String
→ AI Client"]

13. OAuth 2.0 Authorization Flow

How ChatGPT authenticates with the MCP server via OAuth 2.0 (streamable-http transport).

sequenceDiagram
    participant GPT as ChatGPT
    participant NG as ngrok
    participant MCP as MCP Server
    participant OA as SimpleOAuthProvider

    Note over GPT,OA: Phase 1 — Discovery
    GPT->>NG: GET /.well-known/oauth-authorization-server
    NG->>MCP: forward
    MCP-->>NG: OAuth metadata (endpoints, PKCE methods)
    NG-->>GPT: OAuth server metadata

    Note over GPT,OA: Phase 2 — Authorization
    GPT->>NG: GET /authorize?client_id=...&redirect_uri=...&code_challenge=...&state=...
    NG->>MCP: forward
    MCP->>OA: authorize(client, params)
    OA->>OA: Generate auth code (5-min expiry)
    OA->>OA: Store code + code_challenge
    OA-->>MCP: Redirect URL with ?code=...&state=...
    MCP-->>NG: 302 Redirect
    NG-->>GPT: Auth code via redirect

    Note over GPT,OA: Phase 3 — Token Exchange
    GPT->>NG: POST /token (code, code_verifier, client_id, client_secret)
    NG->>MCP: forward
    MCP->>OA: exchange_authorization_code()
    OA->>OA: Validate code + PKCE verifier
    OA->>OA: Generate access_token (1hr) + refresh_token
    OA->>OA: Delete auth code (single-use)
    OA-->>MCP: OAuthToken
    MCP-->>NG: access_token, refresh_token, expires_in: 3600
    NG-->>GPT: Tokens

    Note over GPT,OA: Phase 4 — API Calls
    GPT->>NG: POST /mcp + Authorization: Bearer token
    NG->>MCP: forward
    MCP->>OA: load_access_token(token)
    OA-->>MCP: Valid (not expired)
    MCP->>MCP: Execute tool (same path as STDIO)
    MCP-->>NG: Tool result
    NG-->>GPT: Response

    Note over GPT,OA: Phase 5 — Token Refresh (after 1 hour)
    GPT->>NG: POST /token (grant_type=refresh_token, refresh_token=...)
    NG->>MCP: forward
    MCP->>OA: exchange_refresh_token()
    OA->>OA: Rotate both tokens, revoke old access token
    OA-->>MCP: New OAuthToken
    MCP-->>NG: New access_token + refresh_token
    NG-->>GPT: Refreshed tokens
Key: PKCE (Proof Key for Code Exchange) is enforced by the MCP SDK before code exchange. Auth codes are single-use with 5-minute expiry. Access tokens expire after 1 hour; refresh tokens are perpetual but rotated on each use. The server auto-approves authorization (no login form) — this is a single-user personal server.
